Three Architectural Approaches — Which Direction?

All three use the same medallion + markdown wiki foundation. The difference is in the Gold layer's structured sidecar — what powers the visualization and provenance chain.

A

Markdown-only Gold

Wiki pages + timeline.md + sources.json — no structured sidecar

GOLD ARTIFACTS
📄 wiki/ — markdown pages with inline (source: uuid)
📄 timeline.md — human-readable event list
📄 sources.json — uuid → bronze file path

Strengths

  • Zero new infrastructure
  • LLM chat works immediately
  • Fully human-readable
  • Buildable today, solo

Limits

  • Data engineer has nothing queryable for viz
  • Timeline filtering requires LLM read of all files
  • Provenance is text-only, not clickable
  • Dead-end for dashboard v2

Best for: solo concept validation only. You'll hit the wall fast once the data engineer needs structured data.

B ★

Markdown + JSONL Quads Sidecar (Recommended)

Wiki pages + quads.jsonl + sources_index.json — structured but zero infrastructure

GOLD ARTIFACTS
📄 wiki/ — markdown pages, inline citations
📄 blackboard/quads.jsonl — one quad per line:
{"date","subject","relation","object","source_uuid","confidence","status"} 📄 blackboard/sources_index.json — uuid → bronze path + silver path + excerpt highlight
📄 blackboard/dark_matter.md — auto-generated gap dossier

Strengths

  • Data engineer can query JSONL immediately (Python/pandas/DuckDB)
  • Provenance chain: every citation resolves to Bronze via sources_index
  • JSONL → SQLite migration is one script when ready
  • Dark matter detection: scan for date gaps in quads
  • Confidence = citation count: automatic from duplicate quad detection
  • No running services required

Limits

  • Cross-entity queries (all events by actor X) require file scanning
  • No deduplication logic built in — needs a script
  • Will need migration to SQLite at scale (100+ source files)

Best for: this team, this prototype. Zero infrastructure overhead now, clear migration path later.

C

Markdown + SQLite from Day One

Wiki pages + blackboard.db — invest slightly more up front, avoid migration later

GOLD ARTIFACTS
📄 wiki/ — markdown pages, inline citations
🗄️ blackboard/blackboard.db — SQLite with three tables:
quads(id, date, subject, relation, object, source_uuid, confidence, status) sources(uuid, title, date, source_type, bronze_path, silver_path, excerpt) actors(id, name, role, affiliation)

Strengths

  • SQL queries from day one — data engineer familiar territory
  • Cross-entity queries are trivial (JOIN quads on actor)
  • Provenance is fully relational — citations are foreign keys
  • Dark matter = SQL gap detection query
  • No migration needed later

Limits

  • Schema must be defined before first ingest — brittle if design changes
  • Not human-readable (git diff on .db is useless)
  • One more artifact for LLM ingest script to maintain
  • Adds setup time before first useful output

Best for: if your data engineer prefers SQL and wants to start building dashboard queries in parallel with wiki ingest from day one.

PROVENANCE CHAIN — ALL THREE OPTIONS
Dashboard citation badge → source_uuid → sources_index.json / sources table → Silver markdown excerpt + highlight → Bronze file link
Every claim in the wiki traces back to a specific passage in a specific raw document. Clickable at every step.

Click your preference, or tell me in the terminal. Option B is my recommendation — but if your data engineer lives in SQL, Option C avoids a migration and might be worth the up-front setup.
