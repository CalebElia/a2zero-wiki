Three Architectural Approaches

All three share the same medallion structure and markdown wiki. The difference is in what structured sidecar the Gold layer produces — and how queryable it is for your data engineer.

A
Markdown-Only Gold
Wiki pages + timeline.md + sources.json — no structured sidecar
Gold Artifacts
📄 wiki/ — markdown pages with inline (source: uuid) citations
📄 timeline.md — human-readable dated event list
📄 sources.json — maps uuid → bronze file path
STRENGTHS
  • Zero new infrastructure
  • LLM chat works immediately
  • Fully human-readable
  • Buildable today, solo
LIMITS
  • Data engineer has nothing queryable for viz
  • Timeline filtering requires LLM scan of all files
  • Provenance is text-only, not clickable structure
  • Dead-end for dashboard v2
Best for solo concept validation only. You'll hit the wall fast once the data engineer needs structured data.
B
Markdown + JSONL Quads Sidecar
Wiki pages + quads.jsonl + sources_index.json — structured but zero infrastructure
★ RECOMMENDED
Gold Artifacts
📄 wiki/ — markdown pages with inline citations
📄 blackboard/quads.jsonl — one quad per line:
   {"date", "subject", "relation", "object", "source_uuid", "confidence", "status"}
📄 blackboard/sources_index.json — uuid → bronze path + silver path + excerpt highlight
📄 blackboard/dark_matter.md — auto-generated gap dossier
STRENGTHS
  • Data engineer queries JSONL with Python/pandas/DuckDB today
  • Full provenance chain — every citation resolves to Bronze
  • JSONL → SQLite migration is one script when ready
  • Dark matter detection: scan for date gaps in quads
  • Confidence = citation count, automatic from duplicates
  • No running services required
LIMITS
  • Cross-entity queries (all events by actor X) require file scanning
  • Deduplication logic needs a helper script
  • Migration to SQLite needed at scale (100+ source files)
Best for this team and this prototype. Zero infrastructure overhead now, clean migration path later. The data engineer can start building dashboard queries the day the first ingest runs.
C
Markdown + SQLite from Day One
Wiki pages + blackboard.db — invest slightly more upfront, avoid migration later
Gold Artifacts
📄 wiki/ — markdown pages with inline citations
🗄️ blackboard/blackboard.db — SQLite with three tables:
   quads(id, date, subject, relation, object, source_uuid, confidence, status)
   sources(uuid, title, date, source_type, bronze_path, silver_path, excerpt)
   actors(id, name, role, affiliation)
STRENGTHS
  • SQL queries from day one — familiar to data engineers
  • Cross-entity queries trivial (JOIN quads on actor)
  • Provenance is relational — citations are foreign keys
  • Dark matter = one SQL gap detection query
  • No migration needed later
LIMITS
  • Schema must be defined before first ingest — brittle if design changes
  • Not human-readable (git diff on .db is useless)
  • LLM ingest script must maintain both markdown + DB simultaneously
  • Adds setup time before first useful output
Best if your data engineer prefers SQL and wants to run JOIN queries across actors and quads from week one. Worth asking them before deciding.
