# Strategy Page Foundation / Progression Split

## Fixing Compounding Forgetting in Strategy Synthesis

*Spec: 2026-07-01. Pre-implementation. Addresses Issue 1 of [docs/architecture/2026-06-30-content-quality-audit.md](2026-06-30-content-quality-audit.md).*

---

## The Problem This Solves

Every strategy page (`wiki/strategies/strategy-N-*.md`) is currently a single undifferentiated prose body, fully regenerated by an LLM on every source ingest (`pipeline/pass1b_synthesize.py::_write_synthesis`). The regeneration is supposed to be additive — the Writer prompt explicitly says "preserve all prior facts" — but that instruction only fires when the full prior body is injected as context, and that only happens on the very first ingest (before `wiki/digest.md` exists). Every ingest since has fed the Writer only the compressed digest, and the write path performs an unconditional full-body replace regardless. Net effect: CAP-2020's foundational content (targets, cost estimates, dominant mechanisms) was silently overwritten during the Year 3 ingest and is now absent from every strategy page and, downstream, from the digest itself.

This is a structural problem, not a prompt-tuning problem. Prompt instructions are advisory; an LLM regenerating "a complete replacement" from a lossy summary will drift over enough ingest cycles no matter how the prompt is worded. The fix needs to make forgetting **structurally impossible** for the content that must never change, while still allowing genuine year-over-year synthesis to compound correctly for the content that's supposed to evolve.

---

## The Architecture

Split every strategy page body into two named sections:

```markdown
---
<frontmatter unchanged — synthesis: block still written by Phase C>
---

## Foundation

<Frozen prose. CAP-2020's original design intent for this strategy: the target,
the cost estimate, the dominant mechanism, the theory of change. Written once,
during a one-time backfill (see Migration below). No pipeline pass — Pass 1B,
Phase C, or any future pass — is ever allowed to touch this section again.>

## Progress Synthesis

<LLM-regenerated prose. Synthesizes what has actually happened across every
ingested annual report toward the Foundation's goals — programs launched,
milestones hit, dollar figures, deviations from plan. Regenerated each ingest
by reading the FULL existing Progress Synthesis text (not the digest, not a
summary) plus the new source, so facts accumulate rather than compress.>
```

**Why this beats a validator-only approach:** a "locked facts" validator (checking that key facts are still present after regeneration) is insurance against LLM unreliability — useful, but still probabilistic. A structurally frozen section that literally never enters an LLM's write path has zero probability of loss, by construction. No validator is needed to protect the Foundation section specifically, because nothing can touch it. A validator remains a reasonable future addition for the Progress Synthesis section if drift is observed there, but is out of scope for this spec — revisit if needed after this fix ships.

## Pipeline Changes

### `pipeline/pass1b_synthesize.py`

**New helper — section-aware read/write:**

```python
_SECTION_RE = re.compile(
    r"^##\s*Foundation\s*\n(.*?)(?=^##\s*Progress Synthesis|\Z)"
    r"|^##\s*Progress Synthesis\s*\n(.*?)\Z",
    re.DOTALL | re.MULTILINE,
)

def _split_strategy_sections(body: str) -> tuple[str | None, str | None]:
    """Return (foundation_text, progress_text). Either may be None if the
    page predates the split (legacy single-body page)."""
    foundation = None
    progress = None
    fm = re.search(r"^##\s*Foundation\s*\n(.*?)(?=^##\s*Progress Synthesis\s*\n|\Z)",
                    body, re.DOTALL | re.MULTILINE)
    if fm:
        foundation = fm.group(1).strip()
    pm = re.search(r"^##\s*Progress Synthesis\s*\n(.*)\Z", body, re.DOTALL | re.MULTILINE)
    if pm:
        progress = pm.group(1).strip()
    return foundation, progress


def _assemble_strategy_body(foundation: str, progress: str) -> str:
    return f"## Foundation\n\n{foundation}\n\n## Progress Synthesis\n\n{progress}\n"
```

**Modify `_write_synthesis`** (currently line 514–528) — replace the unconditional full-body overwrite with a section-aware write:

```python
for sb in result.get("strategy_bodies", []):
    strat_path = Path(wiki_root) / (sb["slug"] + ".md")
    if not strat_path.exists():
        print(f"[holistic] WARNING: strategy stub missing: {strat_path} — skipping")
        continue
    existing = strat_path.read_text(encoding="utf-8")
    existing_body = re.sub(r"^---\n.*?\n---\n", "", existing, flags=re.DOTALL).strip()
    foundation, _ = _split_strategy_sections(existing_body)

    if foundation is None:
        # Page predates the split, or Foundation hasn't been backfilled yet.
        # Do NOT silently proceed with a full overwrite — that's the exact bug
        # this spec fixes. Refuse loudly so a human runs the migration first.
        raise RuntimeError(
            f"{strat_path} has no Foundation section. Run the one-time "
            f"Foundation backfill (see migration script) before ingesting."
        )

    new_body = _assemble_strategy_body(foundation, sb["body"])
    _replace_wiki_page_body(str(strat_path), new_body)
    print(f"[holistic] Progress Synthesis updated: {strat_path.name}")
```

**Modify the Writer's context-injection logic** (currently line 340–375): always inject the *existing Progress Synthesis text* (not the digest, not the old whole-body fallback) as `[EXISTING PROGRESS SYNTHESIS]`, regardless of whether `digest_content` is present. The digest continues to be injected too, for cross-strategy narrative awareness — this is additive, not a replacement of the digest mechanism:

```python
existing_progress: dict[str, str] = {}
strategies_dir = Path(wiki_root) / "strategies"
if strategies_dir.exists():
    for strat_file in sorted(strategies_dir.glob("*.md")):
        content = strat_file.read_text(encoding="utf-8")
        body = re.sub(r"^---\n.*?\n---\n", "", content, flags=re.DOTALL).strip()
        _, progress = _split_strategy_sections(body)
        if progress:
            existing_progress[f"strategies/{strat_file.stem}"] = progress

if existing_progress:
    lines = [
        "\n\n[EXISTING PROGRESS SYNTHESIS — every strategy]",
        "READ-UNDERSTAND-INTEGRATE: this is the full prior Progress Synthesis text",
        "for each strategy, not a summary. Preserve every fact in it. Add new depth",
        "from THIS source. Do not discard anything — the Foundation section (not",
        "shown here, and NOT yours to write) already holds the strategy's original",
        "design intent; your job is only to extend the progress narrative.\n",
    ]
    for slug, text in sorted(existing_progress.items()):
        lines.append(f"--- {slug} ---\n{text}\n")
    lines.append("[END EXISTING PROGRESS SYNTHESIS]")
    integration_block += "\n".join(lines)
```

**Modify the system prompt** (STRATEGY BODY RULES, currently line 58–70): clarify that `sb["body"]` in the LLM's JSON response is Progress-Synthesis-only content — it must never attempt to restate or replace Foundation-level material (targets, cost estimates, original mechanism) since that's supplied separately and frozen.

### `pipeline/phase_c_synthesize.py`

No change needed to `write_strategy_synthesis` — confirmed by inspection it only ever touches the `synthesis:` frontmatter block, never the body. It already does exactly the right thing.

**Small separate fix** (same audit, different root cause): `build_strategy_synthesis`'s prompt (line 119–145) only receives a flat entity inventory with no temporal/per-source information, so `year-over-year-arc` has no basis to describe an actual trajectory and defaults to boilerplate. ## Migration — Building Foundation Content

**Revised 2026-07-01.** The original plan recovered Foundation text from git history (commit `1f027a6`, the last known-good state before the Year 3 loss). That approach was rejected: `1f027a6` postdates the Year 2 ingest, so its strategy bodies are already a CAP-2020 + Year 1 + Year 2 blend — recovering it doesn't avoid the disentangling problem, it just moves it earlier, and still requires the same human judgment call to separate "original design intent" from "progress narrative that happened to live in the same paragraph."

**Better approach: derive Foundation directly and only from `wiki/sources/cap/cap-2020.md`.** Foundation is defined as CAP-2020's original design intent — so it should come from CAP-2020 alone, not from any blended wiki state. This is verified structurally sound:

```
Line 407:  ## Strategy 1: Power Our Electrical Grid with 100% Renewable Energy
Line 416:  Combined, these four actions are projected to reduce community-wide
           greenhouse gas emissions by 41% and cost just over $4,100,000. [Source: Page 21]
Line 745:  ## Strategy 2: ...
Line 755:  Combined, these six actions are estimated to reduce community-wide
           emissions by 23% and cost just over $143,000,000. [Source: Page 31]
Line 1258: Strategy 3 — 13.4% / $14,500,000 [Source: Page 45]
Line 2052: Strategy 4 — 8% / $901,000,000 [Source: Page 69]
Line 2649: Strategy 5 — 0.3% / $45,000,000 [Source: Page 85]
Line 3035: Strategy 6 — 0.1% / $7,500,000 [Source: Page 99] (different phrasing
           than 1-5's "Combined, these N actions..." pattern — "In total, these
           actions contribute...")
Line 3472: ## Other Actions   (maps to the wiki's "Strategy 7" — equity/engagement/
           community grant/internal carbon price. No numbered "Strategy 7" heading
           exists in CAP-2020.)
Line 3481: Corrected 2026-07-01 — this spec originally claimed "Other Actions" has
           no quantified target. That was wrong; verified directly against source:
           "Combined, these four actions equate to just under 14% of community-wide
           emissions reductions and are estimated to cost just under $6,000,000.
           [Source: Page 113]" All 7 strategies have an explicit combined target/
           cost figure in CAP-2020 — there is no target-less strategy after all.
```

Six of seven strategies have clean, page-cited, deterministically-locatable target/cost statements right at the top of their section — a bounded, structured extraction task, not an open-ended "read 4,283 lines and write a complete synthesis" task (which is the failure mode this whole spec exists to avoid). Strategy 7 genuinely has no numeric target in CAP-2020 — the Foundation section for it must say so, not invent one.

**Per-strategy extraction is scoped to that strategy's own CAP-2020 section only** (a few hundred to ~1,500 lines, not the full document), further reducing drift risk versus a whole-document pass. Extraction pulls: the combined target %/cost sentence (verbatim, page-cited), the dominant mechanism/policy tool (e.g. Community Choice Aggregation for Strategy 1), and the named actions/initiatives list for that strategy — all facts that exist explicitly in the source text, not narrative the LLM has to synthesize from scratch.

This migration also resolves the CAP-2020-sourced items in Issue 4 of the audit (the original 41%/$4.1M/CCA content, and — pending a targeted look — SolSmart Silver if it's CAP-2020-sourced rather than Year-1-only) as part of the same operation, sourced correctly this time.

## Decisions (resolved 2026-07-01)

1. **Foundation-derived synthesis field — YES.** `build_strategy_synthesis`'s entity inventory prompt will also receive the Foundation text so the `synthesis:` dict (and downstream digest) can cite original targets/mechanisms alongside progress, not just progress alone.
2. **Future CAP revisions — OUT OF SCOPE.** Foundation is frozen relative to the current single foundational document (CAP-2020). A future CAP revision (e.g. CAP-2030) would require a deliberate, human-triggered Foundation re-freeze — no automatic handling needed until that source type exists.
3. **Section-split marker — plain prose, no `##` headers from the LLM.** The Writer returns Progress Synthesis as plain prose only; the pipeline code (not the LLM) wraps it with the `## Foundation` / `## Progress Synthesis` headers on write. This sidesteps the regex-robustness risk entirely — the LLM has no opportunity to emit a stray heading inside its own output. `_validate_synthesis_output` will reject any `sb["body"]` containing a line starting with `##`.
