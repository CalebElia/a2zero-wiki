A wiki that compounds knowledge
across every source it reads.

A2Zero Wiki ingests Ann Arbor's climate planning documents into a structured, AI-queryable knowledge base. Each new source is integrated against what the wiki already knows — not processed in isolation. This is the architectural map of how that happens, end to end.

For Grapevine
The idea

Documents should make the system smarter, not just bigger.

The foundational insight behind LLM-Wiki (Karpathy, 2025) is that the wiki is both output and memory. Most ingest pipelines treat sources as documents to file away. We're building one that reads each new source against everything it has already learned.

Document indexer

Each source is its own island

The pipeline reads Year 3 as if Year 1 and Year 2 never happened. The same initiative under slightly different names produces three disconnected pages. Knowledge accumulates as files, not as understanding. This is what we had after our Year 2 ingest — three flagship initiatives had fragmented across multiple pages.

Compounding wiki

Each source builds on the last

The pipeline reads a curated synthesis of what the wiki already knows before reading the new source. It enters every document already aware of the entities, the year-over-year arcs, the unresolved questions. New facts integrate; name variants resolve to canonical pages; contradictions surface. The wiki becomes smarter with each ingest.

Phase A

Extraction

Four sequential passes turn a single source document into structured entity pages. The new Comprehend step reads the wiki's digest first so the LLM enters the source with full awareness of what's already known.

Pass 0 · copy source & inject YAML Pass 1 · Comprehend → integration plan Pass 1.5 · alias resolution (fuzzy match) Pass 2 · LDP chunked extraction Pass 3 · index rebuild + alias seeding
python -m pipeline.run_ingest source · produces entity pages in wiki/actors/, wiki/initiatives/, etc.
Phase B

Review

Human gate

Three lint modes surface problems in the freshly extracted entity layer. A human reviews the proposals — merge, keep separate, defer — and --apply executes the approved actions. This gate cannot be automated; some near-duplicate decisions require editorial judgment.

--structural · type mismatch, broken links --semantic · near-duplicate detection --backlink · missed entity mentions human annotates review-queue.md --apply
python -m pipeline.lint_wiki · produces merge proposals in review-queue.md
Phase C

Synthesis rebuild

New architecture

Reads the clean, reviewed entity layer and builds the knowledge hierarchy upward: strategy synthesis sections (L1) → cross-strategy narrative and entity map (L2). Runs after lint, never during extraction — synthesis must encode a state a human has signed off on, or it propagates errors into the next ingest.

rebuild strategy synthesis: sections write cross-strategy narrative generate entity map + recent delta writes wiki/digest.md
python -m pipeline.synthesize_wiki · to be built
Phase D

Ready for next ingest

The digest now reflects the post-lint, post-review state of the wiki. The next ingest's Comprehend pass reads it first — entering the new source with full awareness of prior knowledge. The compounding-knowledge property is restored.

digest.md is the post-review snapshot next Comprehend reads digest first knowledge compounds across sources

The cycle closes here. The digest produced in Phase C feeds directly into Phase A of the next ingest. Without this loop, each source is processed in isolation. With it, every new document is integrated against everything the wiki has already learned and reviewed.
Parallel track — blackboard quads. Alongside the wiki, the pipeline can extract structured fact triples (subject / predicate / object) into blackboard/quads.jsonl. This enables graph-structured queries and feeds future analytics. Currently paused pending schema design; the wiki track is the active focus.
Knowledge hierarchy

Three layers, each built from the one below.

The wiki is structured like a pyramid. Entity pages are the broad factual base. Strategy pages aggregate them into community-level synthesis. The digest sits on top — a compressed briefing that fits in an LLM context window and feeds the next ingest.

The factual record

The GraphRAG parallel

Why this works without community detection

Microsoft's GraphRAG (2024) answers global questions about large corpora by building community summaries from entity graphs. The trick is that you can't read every document for every query — you read the community summaries instead. Our L0→L1→L2 hierarchy does the same thing.

Our domain advantage

Strategies replace Leiden clustering

GraphRAG runs Leiden algorithm to discover clusters. We don't have to — Ann Arbor's plan defined them. The 7 A2Zero strategies are semantically grounded, stable across ingests, human-legible, and already embedded in every entity page's related-strategies: field. Domain structure beats algorithmic structure.

In Obsidian

What colleagues will see in the graph view.

The wiki is a native Obsidian vault. Every page is a node. Wikilinks between pages are edges. The graph view shows the knowledge structure as it accumulates — and as the synthesis hierarchy gets built, new structural nodes appear at the top.

The graph as a living artifact

L0 entity pages form dense clusters around their strategies. The seven strategy pages appear as high-connectivity hubs that everything else routes through. Flagship initiatives like Bryant Decarbonization, the Sustainable Energy Utility, and Solarize Ann Arbor sit as cluster centers, linking actors, funding events, locations, and political events.

Once Phase C is built, wiki/digest.md appears as a new top-level node above the strategies — the synthesis pinnacle of the hierarchy.
